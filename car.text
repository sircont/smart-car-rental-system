ABSTRACT
================================================================================

Vehicle rental businesses depend on accurate availability, clear pricing, and
reliable records of bookings and payments (Connolly & Begg, 2015). Manual or
fragmented processes often cause long counter queues, miscommunication, and
errors such as conflicting reservations.

This project designs and implements a web-based Smart Car Rental System that
allows customers to register, browse vehicles, check availability for date
ranges, and manage bookings; enables staff to support maintenance and vehicle
check-in/check-out workflows; and provides administrators with control over
fleet data, branches, users, staff, bookings, payments visibility, and
reporting. Comparable web-based rental prototypes have been reported in the
literature (Thakur, 2021).

The front end uses HTML5, CSS3, JavaScript, and Bootstrap 5 (Bootstrap
contributors, n.d.; Mozilla, n.d.), with AJAX for real-time availability checks.
The back end is implemented in PHP 8.2 using object-oriented modules (PHP Group,
n.d.), with MySQL for persistent storage (Oracle Corporation, n.d.). Security
measures include bcrypt password hashing (OWASP Foundation, n.d.-b; National
Institute of Standards and Technology, 2017b), session-based authentication
(Barth, 2011; National Institute of Standards and Technology, 2017a), CSRF
protection (OWASP Foundation, n.d.-a), input validation, and prepared SQL
statements to mitigate injection risks (MITRE, n.d.; OWASP Foundation, 2021).

The result is a deployable prototype suitable for demonstration and extension
(e.g. live payment gateways and SMS/email). The work concludes with limitations
and recommendations for production hardening and automated testing
(Sommerville, 2016).

Keywords: car rental, web application, PHP, MySQL, role-based access, booking

CHAPTER ONE
INTRODUCTION
================================================================================

1.0 Background to the Study
---------------------------

Urban mobility and short-term vehicle use drive demand for car rental
services. Customers expect to compare vehicles, see daily rates, and reserve a
unit for specific dates without repeated phone calls. Operators must maintain
a single coherent view of which vehicles are available, booked, under
maintenance, or returned—requirements consistent with sound relational data
management practice (Connolly & Begg, 2015; Elmasri & Navathe, 2016).

Web-based information systems address these needs by offering self-service
portals, centralized databases, and role-based access so customers, operational
staff, and administrators interact with appropriate functions only (Fielding
et al., 2022). For small and medium enterprises, stacks such as PHP with MySQL
remain practical because of wide hosting support and rapid delivery of
form-driven applications (PHP Group, n.d.; Oracle Corporation, n.d.; Pressman &
Maxim, 2019).

This project implements a Smart Car Rental System: a customer portal for
discovery and booking, a staff portal for maintenance and handover-related
tasks, and an admin portal for configuration, user management, and reporting.

1.1 Statement of the Problem
----------------------------

Traditional rental desks often rely on manual registers, spreadsheets, or ad
hoc communication. That approach tends to:

  - Increase waiting time and counter congestion.
  - Raise the risk of human error in recording dates, vehicle IDs, and details.
  - Make availability hard to verify in real time, increasing double-booking risk.
  - Provide weak oversight for administrators who need revenue and fleet visibility.

Prior work on web-based car rental systems highlights the value of automating
manual rental procedures and centralising booking data (Thakur, 2021). The
problem addressed here is the need for an integrated, web-based rental system
that enforces structured booking data, separates roles, and applies baseline
web security practices aligned with widely cited risk guidance (OWASP
Foundation, 2021; Awan et al., 2016).

1.2 Goals
---------

The goal is to design and implement a secure, multi-role car rental web
application backed by a relational database, supporting workflows from browsing
to booking and administrative control (Sommerville, 2016).

1.2.1 Specific Objectives
-------------------------

i.   To identify and document functional and non-functional requirements for
     customers, staff, and administrators (Pressman & Maxim, 2019).

ii.  To design a relational data model for users, vehicles, branches,
     bookings, and related entities (Silberschatz et al., 2019).

iii. To develop the system using PHP (OOP), MySQL, and a responsive front end
     with AJAX where appropriate (PHP Group, n.d.; Oracle Corporation, n.d.;
     Mozilla, n.d.).

iv.  To test core modules manually and document results.

v.   To package the system for local or server demonstration (e.g. XAMPP).

1.3 Significance of Study
-------------------------

The work demonstrates three-tier web development, SQL integrity, and security
hygiene for academic purposes (Sommerville, 2016; Connolly & Begg, 2015). For
small rental businesses it offers a prototype extendable with live payments,
notifications, and operational policies, in line with patterns seen in online
service and mobile-commerce research (Garzon et al., 2019; Tan & Ooi, 2018).

1.4 Scope of the Study
----------------------

In scope:
  - Customer self-service: registration, login, password recovery, browsing,
    availability API, booking, profile.
  - Staff: maintenance, bookings view, checklist / check-in-out as implemented.
  - Admin: vehicles, categories, brands, branches, users, staff, bookings,
    payments view, contacts, testimonials moderation, activity logs, reports.
  - Interface accessibility may be assessed against established web guidelines
    (World Wide Web Consortium, 2025).

Out of scope:
  - Full legal/compliance certification, exhaustive load testing, native
    mobile apps, complete production API configuration without credentials.

1.5 Limitations
---------------

  - Time and resources constrain advanced third-party integration.
  - Testing is primarily manual rather than a full automated suite (Pressman &
    Maxim, 2019).
  - Internet and device access are assumed; offline-first use is not primary.

1.6 Structure of Dissertation
------------------------------

Chapter Two reviews literature. Chapter Three describes methodology and tools.
Chapter Four presents implementation. Chapter Five concludes with
recommendations. References and appendix follow.

================================================================================
CHAPTER TWO
LITERATURE REVIEW
================================================================================

2.1 Introduction
----------------

Electronic booking and e-commerce patterns have reshaped how consumers reserve
services (Garzon et al., 2019). Car rental fits this pattern: catalog
presentation, reservation, and order or booking tracking (Tan & Ooi, 2018).

2.2 Evolution of Online Booking
-------------------------------

The web enabled asynchronous self-service and 24/7 access, grounded in standard
client–server semantics (Fielding et al., 2022). Mobile adoption increased
expectations for responsive interfaces and quick feedback—for example,
availability without full page reloads (Tan & Ooi, 2018).

2.3 E-Commerce and Trust
------------------------

Online rental shares traits with e-commerce: product catalog, pricing, and
transaction records. Trust depends on clear totals, secure accounts, and
reliable history (Garzon et al., 2019). Transport-layer security is part of
modern web deployment practice (Rescorla, 2018).

2.4 Technology Trends
---------------------

Server-rendered PHP with MySQL remains common for SME systems (PHP Group, n.d.;
Oracle Corporation, n.d.). AJAX and the Fetch API build on HTTP semantics
(Fielding et al., 2022) and client-side scripting (Mozilla, n.d.). Bootstrap
supports responsive layouts (Bootstrap contributors, n.d.), and Chart.js supports
administrative charts (Chart.js contributors, n.d.). Configuration and API
payloads often use JSON (Bray, 2017).

2.5 Related Work
----------------

Prior systems combine mobile apps, kiosks, or web dashboards (Thakur, 2021).
This project emphasizes a browser-based, role-split rental portal with
server-side validation and SQL constraints (Connolly & Begg, 2015), suitable as
an academic prototype with a path to payment and messaging integration. Broader
verification objectives for web applications are described in industry
standards (OWASP Foundation, 2023).

CHAPTER THREE
METHODOLOGY OF THE STUDY
================================================================================

3.1 Introduction
----------------

Development follows a structured, waterfall-inspired sequence: requirements,
design, implementation, testing, and integration (Sommerville, 2016; Pressman &
Maxim, 2019).

3.2 Functional Requirements
-----------------------------

Customer: Register, authenticate, recover password, browse/search vehicles,
check availability, create and view bookings, manage profile.

Staff: Dashboard; maintenance tasks; bookings; check-in/check-out; checklist.

Admin: Vehicle/category/brand/branch management; users and staff; bookings;
payments view; contacts; testimonials; activity logs; reports.

3.3 Non-Functional Requirements
---------------------------------

Security: Authentication, password hashing, CSRF mitigation, and use of
parameterised queries are consistent with OWASP guidance (OWASP Foundation,
2021, n.d.-a, n.d.-b) and NIST digital-identity recommendations (National
Institute of Standards and Technology, 2017a, 2017b). SQL injection remains a
canonical software weakness class (MITRE, n.d.).

Usability: Responsive UI and clear navigation per portal (Bootstrap
contributors, n.d.; World Wide Web Consortium, 2025).

Maintainability: Separation of public entry scripts, src classes, config, and
database scripts (Pressman & Maxim, 2019).

3.4 Tools and Technologies
--------------------------

Backend: PHP 8.2 (OOP) (PHP Group, n.d.).
Database: MySQL 8.0 (Oracle Corporation, n.d.; Connolly & Begg, 2015).
Frontend: HTML5, CSS3, JavaScript, Bootstrap 5, Chart.js (admin analytics)
(Bootstrap contributors, n.d.; Chart.js contributors, n.d.; Mozilla, n.d.).
Editor: Visual Studio Code (typical).
Environment: Apache, PHP, MySQL (e.g. XAMPP).

3.5 Design Artifacts
--------------------

Data flow diagrams, use case diagrams, and entity-relationship diagrams document
the system (Elmasri & Navathe, 2016; Silberschatz et al., 2019). (Include figures
in final submission.)

3.6 Testing Approach
--------------------

Incremental coding with manual tests for registration, login, booking paths,
role-restricted URLs, and validation messages (Sommerville, 2016).

3.7 Integration
---------------

Shared bootstrap, database connection class, authentication layer, and config
files for app, database, and optional payment/mail/SMS settings. APIs exchange
structured data using JSON where applicable (Bray, 2017; Fielding et al., 2022).

CHAPTER FOUR
SYSTEM IMPLEMENTATION
================================================================================

4.1 Introduction
----------------

This chapter describes realization of the Smart Car Rental System: structure,
data model, interfaces, and testing (Pressman & Maxim, 2019).

4.2 Project Structure
---------------------

public/          Customer pages; admin/ and staff/ sub-areas
src/             Core classes (Database, Auth, Booking, Csrf, Helpers, etc.)
config/          Application and integration settings
database/        Schema and migrations

4.3 Database
------------

MySQL stores users, staff, admin, brands, branches, vehicles, bookings, payments-
related structures, and supporting tables (Oracle Corporation, n.d.). Keys,
unique constraints, and indexes support integrity and queries (Connolly & Begg,
2015; Silberschatz et al., 2019).

4.4 Security Implementation
---------------------------

Passwords are hashed before storage, following password-storage guidance (OWASP
Foundation, n.d.-b; National Institute of Standards and Technology, 2017b).
Sessions rely on HTTP state management mechanisms (Barth, 2011). CSRF tokens on
sensitive forms follow the OWASP CSRF prevention cheat sheet (OWASP Foundation,
n.d.-a). Prepared statements parameterise queries, reducing SQL-injection risk
(MITRE, n.d.; OWASP Foundation, 2021). Production deployments should use TLS
(Rescorla, 2018) and align with HTTP semantics (Fielding et al., 2022).

4.5 Key Features by Role
-------------------------

Customer: Login, register, catalog with filters, AJAX availability, book,
my-bookings, profile, password change, contact/FAQs/feedback as implemented.

Admin: Dashboard with charts, vehicle CRUD, categories/brands/branches,
discounts/insurance, users, staff, bookings, payments, maintenance assignment,
reports, activity log (Chart.js contributors, n.d.).

Staff: Maintenance, bookings, checklist-related screens.

4.6 Backend Logic
-----------------

PHP handles authentication, authorization by role, booking operations, admin
CRUD, and API-style endpoints (e.g. availability) (PHP Group, n.d.).

4.7 Testing
-----------

Per-page and cross-module manual tests; document sample cases in appendix
(Sommerville, 2016).

4.8 Deployment
--------------

Suitable for XAMPP or PHP/MySQL hosting. Production requires HTTPS (Rescorla,
2018), strong secrets, backups, and updated dependencies. Ongoing security
verification can reference structured requirements catalogues (OWASP
Foundation, 2023).

CHAPTER FIVE
CONCLUSION AND RECOMMENDATION
================================================================================

5.1 Conclusion
--------------

The Smart Car Rental System automates core rental workflows through a multi-
role web application and MySQL database, with security-oriented choices grounded
in published guidance (OWASP Foundation, 2021; National Institute of Standards
and Technology, 2017a, 2017b). The project meets its objectives as a functional
prototype for demonstration and further enhancement (Sommerville, 2016; Thakur,
2021).

5.2 Recommendations
-------------------

1. Enable live payment integration (e.g. Flutterwave/Paystack) with verified
   callbacks, building on secure HTTP and API practices (Fielding et al., 2022;
   Rescorla, 2018).

2. Configure SMTP and SMS for confirmations and alerts.

3. Add automated tests (PHPUnit, end-to-end) for booking rules (Pressman &
   Maxim, 2019).

4. Apply rate limiting, security headers, dependency updates, and periodic
   review against ASVS-style criteria (OWASP Foundation, 2023; Awan et al.,
   2016).

5. Conduct user acceptance testing with domain stakeholders.

6. Maintain frequent backups and documented recovery procedures.
